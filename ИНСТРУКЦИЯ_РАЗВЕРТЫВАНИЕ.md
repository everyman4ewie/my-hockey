# Выкладка сайта на Timeweb Cloud (timeweb.cloud) с доменом

Пошаговая инструкция для проекта **Hockey Tactics** (React + Node.js): что подготовить на компьютере, **какие файлы куда копировать**, что **обязательно поменять** перед продакшеном.

---

## 1. Что нужно заранее

| Что | Зачем |
|-----|--------|
| Аккаунт на [timeweb.cloud](https://timeweb.cloud) | Панель облачных серверов |
| **Облачный сервер (VPS)** с Ubuntu 22.04 или Debian 12 | Обычный PHP-хостинг **не подойдёт** — нужен Node.js |
| Минимум **1 GB RAM**, 1 CPU | Для этого приложения достаточно |
| **Домен** (у вас уже есть) | Привязка по A-записи к IP сервера |
| SSH-доступ | Логин `root` или пользователь с `sudo`, IP сервера, пароль или ключ |

---

## 2. Что поменять в проекте **до** выкладки (на своём ПК)

### 2.1. Пароль администратора

Файл `server/admin.json` — поле **`password`**.

- Если оставить `"password": null`, при старте может использоваться пароль по умолчанию из кода сервера (**небезопасно**).
- Укажите свой надёжный пароль в кавычках, сохраните файл.

### 2.2. Логин админа (по желанию)

В том же `server/admin.json` в `profile` можно задать `login`, `email`, `name` — под ваши данные.

### 2.3. Сборка фронтенда

На компьютере в папке проекта:

```bash
cd /путь/к/Hockey
npm install
npm run build
```

Должна появиться папка **`dist/`** с `index.html` и `assets/`.

### 2.4. Локальная проверка (необязательно)

```bash
npm run start
```

Откройте http://localhost:3002 — главная и API должны открываться.

**Менять URL сайта в коде не нужно:** запросы идут на `/api/...` относительно текущего домена.

## 3. Какие файлы **загружать на сервер** (что копировать)

На сервер переносите **только** это (структура важна):

| Куда на сервере | Что копировать с вашего ПК |
|-----------------|----------------------------|
| `.../hockey/dist/` | **Всю папку** `dist/` после `npm run build` |
| `.../hockey/server/` | **Всю папку** `server/` (включая `index.js`, `admin.json`, `data.json`, `tariffs.js`, `tariffLimits.js` и т.д.) |
| `.../hockey/` (корень приложения) | Файлы **`package.json`** и **`package-lock.json`** |

**Не загружайте** (на сервере не нужны для работы сайта):

- `node_modules/` — на сервере ставится заново командой `npm install`
- `src/`, `public/` (как исходники) — в продакшене используется уже собранный `dist/`
- `.git/` — по желанию, для деплоя через git не обязательно

Итоговая структура на сервере (пример):

```
/home/ubuntu/hockey/
├── dist/
│   ├── index.html
│   └── assets/
├── server/
│   ├── index.js
│   ├── data.json          ← база пользователей/планов (создаётся/обновляется на сервере)
│   ├── admin.json
│   ├── tariffs.js
│   └── tariffLimits.js
├── package.json
├── package-lock.json
└── node_modules/          ← появится после npm install
```

## 4. Как **скопировать** файлы на сервер

### Какой IP подставлять вместо `ВАШ_IP`

Используйте **публичный (внешний) IP-адрес вашего облачного сервера** в Timeweb Cloud — тот же, по которому вы заходите по SSH.

| Что это | Откуда взять |
|--------|----------------|
| **Публичный IP** | В панели [timeweb.cloud](https://timeweb.cloud): раздел с вашими серверами → карточка созданного VPS → обычно указано как **IP**, **IPv4** или **Публичный адрес** (формат вида `185.xxx.xxx.xxx` или `89.xxx.xxx.xxx`). |
| **Не подходит** | `127.0.0.1` / `localhost` — это «вы сами», не сервер. IP вашего домашнего роутера из интернета — тоже не то. **Внутренний** IP из частной сети Timeweb (если где-то показан отдельно) для копирования с домашнего компьютера обычно **не используют** — нужен именно адрес, по которому сервер доступен из интернета. |

Команды `scp` и `ssh` выполняются **на вашем компьютере** (Mac/Linux/WSL). Они подключаются **к серверу** по этому публичному IP.

**Пользователь в команде** (`root@` или `ubuntu@`) должен совпадать с тем, под кем вы реально входите по SSH. Часто у нового VPS логин **`root`** или выданный в панели Timeweb (например **`ubuntu`**). Путь на сервере (`/root/hockey` или `/home/ubuntu/hockey`) должен быть тем же пользователем, у которого есть права на запись.

Сначала на сервере один раз создайте папку (по SSH):

```bash
ssh root@ВАШ_IP
mkdir -p /root/hockey
exit
```

(Если используете пользователя `ubuntu`, замените путь на `/home/ubuntu/hockey` и в `scp` укажите `ubuntu@ВАШ_IP:...`.)

### 4.1. Вариант A — SCP (Mac/Linux, из папки проекта Hockey)

```bash
cd /путь/к/Hockey
scp -r dist server package.json package-lock.json root@ВАШ_IP:/root/hockey/
```

Подставьте **вместо `ВАШ_IP`** только цифры публичного IP, без пробелов, например:

```bash
scp -r dist server package.json package-lock.json root@185.123.45.67:/root/hockey/
```

Если при первом подключении спросят про подлинность хоста — введите `yes`. Дальше — пароль от пользователя `root` (или ключа, если настроен вход по ключу).

На **Windows** встроенный `scp` есть в PowerShell и в **WSL**; иначе удобнее вариант B (WinSCP).

### 4.2. Вариант B — SFTP (FileZilla, WinSCP, Cyberduck)

- **Хост:** тот же публичный IP (или домен, если уже привязан к серверу).
- **Порт:** `22` (SSH).
- **Пользователь и пароль:** как для SSH (`root` / `ubuntu` и т.д.).
- Перетащите те же папки **`dist`**, **`server`** и файлы **`package.json`**, **`package-lock.json`** в каталог `hockey` на сервере.

### 4.3. Вариант C — Git

Залить репозиторий на сервер, затем на сервере выполнить `npm install` и `npm run build` (нужен Node.js и dev-зависимости для сборки), либо собирать только локально и загружать готовый **`dist`**, как в п. 3.

## 5. Пошагово на **Timeweb Cloud** и на сервере

### Шаг 1. Создать сервер в панели timeweb.cloud

1. Войти в [timeweb.cloud](https://timeweb.cloud).
2. Создать **облачный сервер** (VPS): ОС **Ubuntu 22.04** или **Debian 12**.
3. Записать **публичный IP** сервера.
4. Сохранить **пароль root** или настроить SSH-ключ.

### Шаг 2. Подключиться по SSH

```bash
ssh root@ВАШ_IP
```

### Шаг 3. Установить Node.js 20

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt update && apt install -y nodejs
node -v
```

### Шаг 4. Загрузить проект в каталог (пример `/root/hockey`)

Скопируйте файлы (см. раздел 4), затем на сервере:

```bash
cd /root/hockey
npm install --production
```

### Шаг 5. Первый запуск (проверка)

```bash
PORT=3002 node server/index.js
```

В браузере: `http://ВАШ_IP:3002` — если открывается, остановите процесс (**Ctrl+C**).

### Шаг 6. PM2 — чтобы сайт работал постоянно

```bash
npm install -g pm2
cd /root/hockey
pm2 start server/index.js --name hockey
pm2 save
pm2 startup
```

Выполните команду, которую выведет `pm2 startup` (для автозапуска после перезагрузки).

### Шаг 7. Nginx — сайт на порту 80/443 с вашим доменом

```bash
apt install -y nginx
nano /etc/nginx/sites-available/hockey
```

Вставьте (замените `example.com` на **ваш домен**):

```nginx
server {
    listen 80;
    server_name example.com www.example.com;

    location / {
        proxy_pass http://127.0.0.1:3002;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
    }
}
```

Включить сайт и перезапустить Nginx:

```bash
ln -sf /etc/nginx/sites-available/hockey /etc/nginx/sites-enabled/
nginx -t
systemctl reload nginx
```

При необходимости откройте firewall:

```bash
ufw allow OpenSSH
ufw allow 'Nginx Full'
ufw enable
```

### Шаг 8. Домен → IP в Timeweb

1. Панель Timeweb (домены) или у регистратора домена.
2. **A-запись** для `@` и для `www` → **IP вашего облачного сервера**.
3. Подождать распространения DNS (от нескольких минут до часа).

Проверка: `http://ваш-домен.ru` должен открывать сайт через Nginx.

### Шаг 9. HTTPS (Let's Encrypt)

```bash
apt install -y certbot python3-certbot-nginx
certbot --nginx -d example.com -d www.example.com
```

Дальше следуйте подсказкам certbot.

## 6. Оплата ЮKassa, файл `.env` и поведение подписки

### 6.1. Файл `.env`

В корне проекта (рядом с `package.json`) создайте файл **`.env`** вручную или скопируйте шаблон:

```bash
cp .env.example .env
```

Файл **`.env`** в git не попадает (указан в `.gitignore`) — секреты и URL не должны оказываться в репозитории. На сервере тот же набор переменных задаётся в `.env` в каталоге приложения или через переменные окружения в PM2.

**Минимум для ЮKassa:**

| Переменная | Назначение |
|------------|------------|
| `PUBLIC_APP_URL` | Публичный URL сайта **без** слэша в конце (например `https://example.com` или локально `http://localhost:5173`). Нужен для возврата пользователя после оплаты. |
| `YOOKASSA_SHOP_ID` | Идентификатор магазина в ЮKassa. |
| `YOOKASSA_SECRET_KEY` | Секретный ключ API (тестовый или боевой). |
| `YOOKASSA_RECEIPT_VAT_CODE` | (Необязательно) Код ставки НДС в позиции чека: `1` — без НДС; иначе см. [документацию ЮKassa](https://yookassa.ru/developers/payment-acceptance/receipts/54fz/yoomoney/parameters-values). |
| `YOOKASSA_TAX_SYSTEM_CODE` | (Необязательно) Код системы налогообложения магазина в чеке, если ЮKassa требует (1 — ОСН, 2 — УСН доход и т.д.). |

**Чеки 54-ФЗ:** если в личном кабинете ЮKassa включена **онлайн-касса / чеки**, без корректного чека платёж отклоняется с ошибкой вроде «Receipt is missing or illegal». В запрос к API передаётся объект `receipt` с **email покупателя** из профиля пользователя — пользователь должен указать корректный email в кабинете. При ошибках по НДС или системе налогообложения подберите `YOOKASSA_RECEIPT_VAT_CODE` и при необходимости `YOOKASSA_TAX_SYSTEM_CODE` под вашу организацию.

После изменения `.env` на сервере перезапустите приложение: `pm2 restart hockey`.

В [личном кабинете ЮKassa](https://yookassa.ru) в настройках магазина укажите URL **HTTP-уведомлений (webhook)**:

`https://ваш-домен.ru/api/payments/yookassa/webhook`

(для локальной отладки без белого IP используйте туннель, например ngrok, и подставьте выданный HTTPS-URL.)

### 6.2. Поведение подписки и ограничения

- В личном кабинете (**Тарифы**) пользователь с тарифом «Про» и включённым автопродлением может **отменить автопродление** — отвязывается сохранённая карта (`yookassaPaymentMethodId`), дальнейшие списания не выполняются; оплаченный доступ к «Про» сохраняется до `tariffExpiresAt`.
- При **неудачном автосписании** сервер периодически **повторяет попытку**, пока дата следующего списания остаётся в прошлом. В интерфейсе это кратко описано в блоке «Если не удалось списать оплату». Отдельная политика (уведомления по email, принудительный даунгрейд) **не реализована** — при необходимости добавляется отдельно.

## 7. Что менять **на сервере** при обновлении сайта (не потерять пользователей)

Зарегистрированные пользователи и данные лежат в **`server/data.json`** на сервере. Настройки админки и страниц — в **`server/admin.json`**. Их **нельзя затирать** пустыми файлами с вашего компьютера.

**Безопасный порядок:**

1. **Перед обновлением** — команды ниже выполняются **на облачном сервере** после входа по SSH (`ssh root@72.56.233.203`), **не** в терминале на Mac/Windows. Путь `/root/hockey` существует **только на VPS**; на домашнем компьютере у вас другой путь к проекту (например `~/Hockey`).

   ```bash
   ssh root@72.56.233.203
   cd /root/hockey
   cp server/data.json server/data.json.bak
   cp server/admin.json server/admin.json.bak
   ```

   Если у вас на сервере другой каталог (например `/home/ubuntu/hockey`), подставьте его вместо `/root/hockey`.

2. Локально у себя: `npm run build` — получите актуальную папку **`dist/`**.

3. **Залить на сервер:**
   - всю папку **`dist/`** (заменить целиком);
   - из **`server/`** — только **изменённые файлы** (`index.js`, `tariffs.js`, и т.д.), **не перезаписывая** рабочие **`data.json`** и **`admin.json`** на сервере.  
     Удобно: `scp` отдельных файлов или SFTP с выбором файлов; при копировании всей папки `server/` сначала **сохраните** на сервере копии `data.json` и `admin.json`, после заливки **верните** их обратно, если они перезаписались.

4. **`package.json` / `package-lock.json`** — обновить, если менялись зависимости, затем на сервере:

   ```bash
   cd /root/hockey
   npm install --production
   ```

5. Перезапуск:

   ```bash
   pm2 restart hockey
   ```

**Кратко:** обновляете **код и `dist/`**, а **`data.json` и `admin.json` на сервере оставляете теми же**, что были до выкладки (если только вы сами не правите админку осознанно).

### 7.1 Автоматическая выгрузка с Mac/Linux (rsync + ssh)

В репозитории есть скрипт `scripts/rsync-to-server.sh`: он собирает `dist/`, копирует код **`server/` без** перезаписи на сервере **`data.json`**, **`admin.json`** и папки **`uploads/`** (видео пользователей), затем на сервере выполняет `npm ci --omit=dev` и `pm2 restart hockey`.

Перед запуском задайте SSH-цель и при необходимости каталог на VPS:

```bash
cd /путь/к/Hockey
export DEPLOY_SSH='root@72.56.233.203'    # ваш пользователь@IP
export DEPLOY_PATH='/root/hockey'         # если отличается от /root/hockey
npm run deploy:server
# или: bash scripts/rsync-to-server.sh
```

Нужны **настроенный SSH-ключ** к серверу и установленные локально `rsync` и `ssh`. Скрипт перед копированием создаёт на сервере резервные копии `data.json` / `admin.json` с меткой времени в имени файла.

Файл **`.env` на сервере скрипт не трогает** — переменные окружения остаются как были.

## 9. Типичные проблемы

| Симптом | Что проверить |
|---------|----------------|
| Белый экран / 502 | `pm2 logs hockey`, `pm2 restart hockey`, есть ли папка `dist/` рядом с `server/` |
| Не открывается по домену | DNS A-запись, прошло ли время; `ping` на домен |
| API не работает | Nginx проксирует **весь** сайт на Node (путь `/api` тот же хост) |
| Забыли пароль админа | Правка `server/admin.json` и перезапуск `pm2 restart hockey` |
| Оплата не создаётся / 503 | Раздел **6**: на сервере есть `.env` с `PUBLIC_APP_URL`, `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY`; после правок — `pm2 restart hockey` |
| Тариф не обновился после оплаты | Webhook в ЮKassa указывает на `https://домен/api/payments/yookassa/webhook`; для localhost нужен туннель |
